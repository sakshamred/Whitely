The problem

You're renting a conditional.

LaunchDarkly charges per seat, per environment, per MAU. Your flag configuration lives on their servers. Their SDK phones home on every evaluation. You are paying a recurring toll to execute an if-statement.

Cloud flag services want you to run their relay agents, configure their webhooks, trust their uptime SLA, and audit their changelog every time the API shifts. You are not a customer — you are a dependency in their business model.

Every flag you create teaches their system your product roadmap. Every evaluation is a data point they didn't earn. And when their service degrades, your flags return stale data because the source of truth is in their data center, not yours.

LaunchWhitely is a binary. It runs wherever you run things. SQLite stores your flags on disk; memory serves them in nanoseconds. Your flags. Your server. Your call.

Architecture

How it works

  • Single binary One statically compiled Rust binary with no runtime dependency. Copy to a server, run it. Starts in under 50ms. Relocate it with scp. Install via Cargo or npm — both deliver the same binary.
  • SQLite-backed Flags persist to an embedded SQLite file. No external database, no connection pool. The entire store is one file you can inspect with any SQLite browser, back up with cp, and migrate with rsync.
  • SSE real-time SDK clients subscribe over Server-Sent Events — plain HTTP, no WebSocket overhead, no polling interval. On connect: a full put snapshot. On change: incremental patch or delete. Keepalive comments every 15s.
  • In-memory eval All active flag configs load into memory at start and stay current via internal events. Evaluation is a hash lookup — zero DB queries per request, zero network latency. SQLite handles durability; RAM handles throughput.
  • XxHash64 bucketing Percentage rollouts use XxHash64 over flagKey + salt + contextKey. Same context key always gets the same variation — deterministic, consistent, and independent of clock or order.

Requires the Rust toolchain. Install via rustup.rs if you don't have it yet. Builds the binary from crates.io.

bash — npm 
# install globally — downloads a pre-built binary
$ npm install -g launchwhitely

# start the server
$ launchwhitely serve

No Rust toolchain required. Downloads a pre-built binary for your platform. Node ≥ 16. Published at npmjs.com/package/launchwhitely.

bash — npx 
# no install — run the latest release directly
$ npx launchwhitely@latest serve

Great for CI pipelines and first-look evaluation. Downloads and caches on first run. No global install needed.

bash — curl 
# download and install (Linux & macOS)
$ curl -fsSL https://launchwhitely.dev/install.sh | sh

# start the server
$ launchwhitely serve

Installs to ~/.local/bin. Set LAUNCHWHITELY_INSTALL_DIR to override, LAUNCHWHITELY_VERSION to pin a version. Review the script at /install.sh before piping to sh.

Zero to flags in 60 seconds

SSE stream → http://localhost:7777/api/sdk/stream

  2. 02

    Create an environment & flag

    bash
    $ export ADMIN_KEY="lw_<paste-key-here>"

# create an environment
$ curl -sSf http://localhost:7777/api/envs \
    -H "Authorization: Bearer $ADMIN_KEY" \
    -H 'Content-Type: application/json' \
    -d '{"key":"production","name":"Production"}'

# create a boolean flag
$ curl -sSf http://localhost:7777/api/flags \
    -H "Authorization: Bearer $ADMIN_KEY" \
    -H 'Content-Type: application/json' \
    -d '{"key":"dark-mode","kind":"bool","variations":[false,true]}'
  3. 03

    Enable the flag & create an SDK key

    bash
    # enable in production
$ curl -sSf -X PATCH http://localhost:7777/api/envs/production/flags/dark-mode/toggle \
    -H "Authorization: Bearer $ADMIN_KEY" \
    -d '{"enabled":true}'

# create a read-only SDK key scoped to production
$ curl -sSf http://localhost:7777/api/keys \
    -H "Authorization: Bearer $ADMIN_KEY" \
    -d '{"role":"sdk","env_key":"production","name":"my-app"}'

$ export SDK_KEY="lw_<paste-sdk-key>"
  4. 04

    Evaluate a flag for a user

    bash
    $ curl -sSf http://localhost:7777/api/eval \
    -H "Authorization: Bearer $SDK_KEY" \
    -d '{"context":{"key":"user-123","attributes":{"plan":"pro"}},"flag_key":"dark-mode"}'
← {"flag_key":"dark-mode","variation":1,"value":true}
  5. 05

    Subscribe to real-time changes

    Keep a connection open. Clients receive flag updates the instant they're written.

    bash — SSE
    $ curl -sSf http://localhost:7777/api/sdk/stream \
    -H "Authorization: Bearer $SDK_KEY" \
    -H "Accept: text/event-stream"

event: put
data: {"type":"put","data":{"env_key":"production","flags":[...],"configs":[...]}}

event: patch
data: {"type":"patch","flag_key":"dark-mode","data":{...updated config...}}

event: delete
data: {"type":"delete","flag_key":"old-flag"}
Flag types

Four kinds of flag

Boolean, string, number, and JSON variations — with weighted rollouts and targeting rules on all of them.

bool
Boolean

On/off switches. The classic feature flag. Use for enabling or disabling features per environment or targeting rule.

"variations": [false, true]
string
String

Serve different text values per user. Config strings, A/B copy, UI labels — anything a string can hold.

"variations": ["control", "treatment-a", "treatment-b"]
number
Number

Rate limits, timeouts, thresholds — any numeric value you'd otherwise hardcode in config. Change it without a deploy.

"variations": [100, 500, 1000]
json
JSON

Serve arbitrary structured data — entire feature config objects, pricing tiers, UI themes. Any valid JSON object.

"variations": [{"theme":"default"}, {"theme":"dark","accentColor":"#00ff88"}]

Security

Built for the paranoid

Rate limiting, TLS, and network isolation are your reverse-proxy's job (nginx, Caddy, etc.).

Hashed key storage

API keys are stored as sha256(raw_key). Raw key material is never persisted to disk. If the database is exfiltrated, keys are still safe.

Constant-time comparison

Auth uses constant-time comparison to prevent timing attacks during key validation. No early exits in the comparison loop.

Zero DB queries per eval

Flag evaluation reads only from in-memory state. The database is never touched during a flag check — zero surface area for SQL injection on eval paths.

Scoped SDK keys

SDK keys are read-only and scoped to a single environment. They cannot write, create, or delete. Admin keys required for any mutation.
